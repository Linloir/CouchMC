# MC Controller — macOS server

Swift / SwiftUI port of the Windows server, mirroring the WinUI 3 app at
the page level: **设备发现 / 设置 / 全局设置 / 关于**. Speaks the same wire
protocol (see [docs/protocol.md](../docs/protocol.md)) and LAN discovery
spec (see [docs/discovery.md](../docs/discovery.md)) as the Windows
server, so an Android client targeting either server is correct by
construction.

## Prerequisites

- macOS 14 (Sonoma) or newer at runtime (deployment target is macOS 14.0).
- Xcode 16 / Swift 6.1 for building. Xcode 17 / Swift 6.2 unlocks the
  Liquid Glass code path (see [Liquid Glass](#liquid-glass) below).
- Java Edition Minecraft installed under any of the standard wrappers
  (the foreground detector matches `java*`/`minecraft*`/`mojang*` in
  the process name + bundle ID).

## Build & run

First-time setup:

```bash
cd mac
# Bundle adb so USB auto-pairing works (optional but recommended).
bash scripts/fetch-adb.sh
# Build + copy to /Applications + register + restart ControlCenter.
bash scripts/install.sh
```

⚠️ **Install to `/Applications` is mandatory, not optional.** Apps
launched directly from `build/Debug/McController.app` work in most
ways but the **menu bar status item gets misplaced** by macOS's
layout engine (it lands behind the MBP-14/16 notch or off-screen at
weird X coordinates) because ad-hoc-signed apps in user paths aren't
treated like proper installed apps by `ControlCenter` (Sequoia+) /
`SystemUIServer` (older). `scripts/install.sh` is the canonical
workflow — it handles the bundle copy, LaunchServices re-register,
and the `killall ControlCenter` step needed to reset the menu bar's
cached layout.

If you opened the Xcode project directly and ran the app from
`build/Debug/`, you'll see the warning card on the Discovery page —
its **Restore Icon** button performs the same reset, but installing
once and using the `/Applications` copy is more reliable than
restoring after every rebuild.

Subsequent rebuilds:

```bash
# install.sh re-runs the whole pipeline: regenerate Xcode project,
# rebuild, copy to /Applications, re-register, ControlCenter
# restart, launch.
bash scripts/install.sh
```

For pure compilation (no install), the lower-level scripts still
work:

```bash
bash scripts/build.sh   # build only; .app stays in mac/build/
```

Or open `McController.xcodeproj` directly in Xcode. The default scheme
builds + launches `McController.app` from the build directory — fine
for code iteration as long as you don't care about the menu bar
icon. On first launch the app prompts for **Accessibility**
permission — without it, CGEvents are silently dropped and MC won't
see input.

### Menu bar item not visible?

If you don't see the white grass-block icon in the menu bar after
launching, you're almost certainly running a menu-bar manager — most
likely **Hidden Bar**, **Bartender**, **iStat Menus**, or **Vanilla**.
These tools auto-hide newly-registered status items by default
(they're moved to negative X coordinates, off-screen).

The icon is still being created — verify with:

```bash
# Lists every NSStatusItem owned by McController; the menu-bar icon
# shows up as a `layer=25 title='Item-0'` row. If X is negative,
# something is hiding it.
swift -e '
import AppKit
import CoreGraphics
let pid = (NSRunningApplication.runningApplications(withBundleIdentifier:
    "cn.linloir.couchmc.mac").first?.processIdentifier).map(Int.init) ?? 0
let info = (CGWindowListCopyWindowInfo([.optionAll], kCGNullWindowID)
    as? [[String: Any]]) ?? []
for w in info where (w[kCGWindowOwnerPID as String] as? Int) == pid {
    print(w[kCGWindowBounds as String] ?? [:])
}'
```

To reveal the icon, click the manager's overflow chevron (the `<`
glyph in the menu bar) and drag the grass-block icon into the
permanently-visible section, or whitelist McController in the
manager's preferences.

## Storage locations

Per Apple conventions:

| File | Purpose |
|---|---|
| `~/Library/Application Support/McController/config.json` | Controller tuning (`ServerConfig`, profiles, bindings) |
| `~/Library/Application Support/McController/appearance.json` | Liquid Glass preference |
| Login-item state | `SMAppService.mainApp` (managed by macOS) |

No registry equivalents — the Windows version's `HKCU\…\Run\McController`
and `%LOCALAPPDATA%\McController\errors.log` map to `SMAppService` and
the system console (`os_log` / Console.app) respectively.

## Liquid Glass

The Global Settings page has a three-way switch (跟随系统 / 开启 / 关闭).
Behavior:

- **跟随系统**: glass surfaces on macOS 26 (Tahoe), traditional materials
  on older systems.
- **开启**: forces glass when supported; falls back to materials on
  older systems.
- **关闭**: never glass — solid card backgrounds with separator borders.

Implementation: `UI/LiquidGlass.swift` exposes `View.glassSurface(_:enabled:)`.
The macOS 26 `.glassEffect(.regular, in: shape)` call is gated behind
`#if compiler(>=6.2)` so the project still builds with Xcode 16
toolchains (which lack the Glass type). When opened in Xcode 17+ the
glass branch automatically activates.

## ADB integration

The bundled adb is staged at `McController/Resources/adb/adb` (run
`scripts/fetch-adb.sh` to populate). The Xcode resources phase copies
the whole `Resources/adb/` directory into the .app bundle, so the
shipped binary is found at `McController.app/Contents/Resources/adb/adb`.

`Services/AdbDiscovery.swift` searches in this order:

1. `<bundle>/Contents/Resources/adb/adb` (preferred — the bundled copy)
2. `<bundle>/Contents/Resources/adb` (flat variant for hand-installs)
3. `$PATH` fallback (so a dev-built debug binary picks up the system
   adb without needing to bundle one)

Auto `adb reverse tcp:<port> tcp:<port>` fires on every newly-connected
USB device, dedup'd by serial so we don't re-forward every 3 s poll.

## Project regeneration

The project file is regenerated from the source tree via
`scripts/gen-xcodeproj.swift`. **Whenever you add or remove a Swift
file, re-run it** (or just run `bash scripts/build.sh`, which does it
for you). Hand-editing the pbxproj is fine but the regenerator is
authoritative — if your edits conflict, the regenerator wins.

UUIDs are derived from MD5 hashes of `(kind, key)` so they're stable
across regenerations. New files get fresh hashes; deleted files'
UUIDs simply stop appearing.

## Differences vs. the Windows server

By design (the user requested feature parity except for one thing):

- **No Acrylic transparency sliders** in Global Settings — the Windows
  version exposed `appearance.json` sliders for the WinUI Acrylic tint;
  macOS replaces that with the Liquid Glass tri-state.
- **No tray "Open Panel" double-click**: macOS menu bar items always
  open their menu on click, so we use a menu item instead of the
  click-to-open shortcut.
- **No `--selftest` CLI flag** yet — the equivalent on macOS would be
  a small headless harness that posts CGEvents at 125 Hz; we'll add it
  when the input feel needs tuning on Mac.

Everything else (3-state mode system, LAN discovery, sub-pixel scaling,
JoystickToWasdMapper hysteresis, ButtonRouter releaseAll-on-disconnect,
profile picker semantics) is a direct port and behaves identically.
